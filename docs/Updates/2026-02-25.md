### 【面向开发者】

#### v0.1.005 🏗️ 工作流 CRUD + 画布引擎 + 三层同步机制
- **后端 CRUD API**（`app/api/workflow.py`）：实现 5 个端点——`GET /api/workflow/`（获取当前用户工作流列表，仅返回元数据不含节点内容）、`POST /api/workflow/`（新建工作流）、`GET /api/workflow/{id}/content`（获取完整 nodes/edges JSON）、`PUT /api/workflow/{id}`（更新工作流，供自动保存调用）、`DELETE /api/workflow/{id}`（删除工作流）。所有路由通过 JWT 中间件鉴权，结合 Supabase RLS 确保用户只能访问自己的数据。
- **Pydantic 模型**（`app/models/workflow.py`）：定义 `WorkflowCreate`、`WorkflowUpdate`、`WorkflowMeta`（不含 nodes/edges 的轻量元数据）、`WorkflowContent`（含完整画布 JSON）四个数据模型。
- **Dashboard 三栏布局**：`layout.tsx`（共享容器）+ `Sidebar.tsx`（左侧边栏，展开 280px / 收起 64px 图标模式，工作流列表导航，右键菜单支持重命名和删除）+ `Navbar.tsx`（顶栏，用户头像 + 新建工作流按钮）+ `MobileNav.tsx`（移动端底部导航栏，仅 md 以下显示）。
- **工作流列表页**（`workspace/page.tsx`）：Server Component SSR 首屏渲染工作流卡片列表，点击跳转 `/workspace/[id]` 画布页，加载中显示骨架屏动画。
- **Zustand Store**（`use-workflow-store.ts`）：集中管理 `nodes`、`edges`、`currentWorkflowId`、`isDirty` 状态，提供 `setNodes`、`onNodesChange`、`updateNodeData`、`markClean` 等操作方法。
- **无限画布**（`WorkflowCanvas.tsx`）：基于 `@xyflow/react` 实现节点拖拽、连线、缩放、平移；移动端适配双指缩放、单指平移、防误触。
- **三层防抖同步**（`use-workflow-sync.ts`）：第一层 UI 变更实时写入 Zustand Store；第二层通过 localforage 以 500ms 防抖写入 IndexedDB 本地缓存；第三层以 3-5s 防抖调用 `PUT /api/workflow/{id}` 同步到云端。云端同步成功后清除 dirty 标记，失败时右上角提示"离线修改已保存本地"。配合 `window.onbeforeunload` 未保存确认弹窗，以及崩溃恢复检测（比较本地和云端的 `updated_at` 时间戳，本地更新则弹窗提示恢复或丢弃）。
- **属性测试**：CRUD 数据往返一致性（Property 4）、用户数据隔离（Property 5）、Store 状态一致性（Property 6）、脏标记管理（Property 7）、崩溃恢复冲突检测（Property 8）。

#### v0.1.006 🤖 多平台 AI 模型路由 + 两段式工作流智能生成
- **YAML 配置中心**（`config.yaml`）：零硬编码地定义 8 大 AI 平台连接信息（阿里云百炼、DeepSeek、月之暗面 Kimi、火山引擎豆包、智谱 AI + 七牛云、优云智算、硅基流动），以及节点类型到模型的路由映射规则（node_routes）、容灾降级链（fallback chains A/B，首选平台故障时按链路自动切换备选）、执行引擎参数（max_nodes、json_validation_retries）。
- **配置加载器**（`config_loader.py`）：递归解析 `$ENV_VAR` 环境变量引用，`lru_cache` 确保应用生命周期内只加载一次。
- **AI 模型路由**（`ai_router.py`）：根据节点类型查询配置中的路由链，确定该调用哪个平台的哪个模型。使用 openai Python SDK 统一适配所有平台（通过不同的 `base_url` + `api_key` 区分）。首选平台超时或报错时，按降级链自动切换到下一个备用平台。格式严格型任务（如 JSON 输出）禁止路由到代理聚合平台。
- **9 种节点类型**（`app/models/ai.py`）：定义 `trigger_input`（输入触发）、`ai_analyzer`（需求分析）、`ai_planner`（流程规划）、`outline_gen`（大纲生成）、`content_extract`（内容提炼）、`summary`（总结归纳）、`flashcard`（闪卡制作）、`chat_response`（对话回复）、`write_db`（数据写入）共 9 种 MVP 节点类型枚举，每种 LLM 节点附带专属 System Prompt 模板和 JSON Schema 输出约束。
- **两段式工作流生成**（`POST /api/ai/generate-workflow`）：用户输入一句自然语言学习目标后，第一阶段由 AI_Analyzer（走路由链 A，当前首选百炼 qwen3-turbo）将其解析为结构化需求 JSON（包含 goal、user_defined_steps、design_requirements、constraints）；第二阶段由 AI_Planner 根据需求 JSON 生成完整的 `{ nodes[], edges[] }` 画布数据，每个 node 包含 id、type、position、data（label、system_prompt、model_route、status）。同时生成 Implicit_Context 暗线上下文（global_theme、language_style、core_outline、target_audience），作为后续所有节点执行时的共享背景信息自动注入。AI 返回的 JSON 通过 Pydantic Schema 强制验证，验证失败自动重试最多 3 次。
- **Prompt 注入防护**：对用户输入进行转义过滤，System Prompt 中设置沙箱指令限制 AI 行为边界，防止恶意输入篡改模型行为。
- **属性测试**：YAML 环境变量解析（Property 9）、节点类型到模型路由正确性（Property 10）、容灾降级链执行（Property 11）、AI 输出 Schema 验证（Property 12）、JSON 验证失败重试（Property 13）、节点类型 Schema+Prompt 完整性（Property 19）、Prompt 注入防护（Property 22）。
