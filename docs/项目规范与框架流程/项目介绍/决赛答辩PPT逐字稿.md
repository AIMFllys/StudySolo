## 答辩前必读 · 备查区（不念，追问时翻阅）

### 仓库与版本

- **Polyglot Monorepo**：`frontend/`（Next.js 16，端口 2037）、`backend/`（FastAPI，2038）、`agents/`（多微服务 8001–8099）、`introduce/`（Vite 介绍页）、`shared/`（Git Submodule）、`supabase/migrations/`。
- **竞赛提交版**冻结于 GitHub `AIMFllys/StudySolo`；**本仓库**为活跃开发分支。

### 前端核心路径

- 画布：`@xyflow/react` + Zustand（`use-workflow-store`），Undo/Redo、执行追踪 UI。
- 画布 → AI：`use-canvas-context.ts` 的 `serialize()` 生成 `CanvasContext`（节点摘要、DAG 文本、选中节点、坐标），由 `SidebarAIPanel` → `useStreamChat` 发送。**不含**全站路由自动感知。
- 三种模式：Chat / Plan / Create；Create 下区分 BUILD（空画布 → `/api/ai/generate-workflow`）与 MODIFY（增量 JSON + 重试）。
- 统一流式入口：`/api/ai/chat`，实现于 `backend/app/api/ai/chat.py`。

### 后端核心路径

- DAG 引擎：`backend/app/engine/` 模块化架构 → `executor.py` / `topology.py` / `node_runner.py` / `level_runner.py` / `loop_runner.py`。
- SSE 事件：`node_input` → `node_status` → `node_token` → `node_done` → `loop_iteration` → `workflow_done`。
- 执行 API：`POST /workflow/{id}/execute`（`backend/app/api/workflow/execute.py`）。
- 节点注册：`backend/app/nodes/__init__.py` 自动发现 → `NODE_REGISTRY`。
- Prompt 体系：`backend/app/prompts/` 模块化 Markdown + 模板变量。

### 节点口径

- 参赛说明书写 **18 种**（冻结版）；当前代码枚举 **20 项**（含 `community_node`、`loop_group`）。口播说 **「20 种节点类型」** 或 **「18 种学习向 + 控制流与社区扩展」** 均可。

### Agent 子后端

- `agents/code-review-agent` (8001): ✅ 已落地，110 passed tests
- `agents/study-tutor-agent` (8004): ✅ 阶段版已落地，35 passed tests

### 数据与安全

- Supabase PostgreSQL + Auth + **RLS**；后台审计。

### 后台诊断、节点测试与容灾

- **一键诊断面板**：`backend/app/api/admin_diagnostics.py` + `frontend/src/app/(admin)/admin-analysis/diagnostics/page.tsx`；并行拨测 5 大类组件（DB / AI 模型 / 子 Agent / 外部 API / 内部服务），输出 `latency_ms`、错误详情、Markdown / JSON 报告一键导出。
- **一键节点测试**：后台管理面板提供**节点级单测入口**，可独立触发任一工作流节点的执行与结果校验，不必跑完整工作流即可回归测试。
- **多模型容灾降级**：`backend/app/services/llm/router.py` 的 `call_llm_structured` 按 `_build_route_candidates(node_type)` 生成候选 SKU 链路，失败自动切备选，流式场景已 yield 则抛 `Streaming interrupted` 避免"半句话切换"；每次失败通过 `record_error_attempt` 写入计费账本（`attempt_index` / `is_fallback` 字段）。
- **自检 SOP**：`docs/项目规范与框架流程/功能流程/系统自检与诊断/` 定义了"一键测试所有 AI 模型/Agents/数据库/内部服务"的标准流程，日志统一落盘 `scripts/logs/`。

## P1 · 开场：一句原话 + 三组信号（约 60s）

**【PPT 提示】**：
- 左上角一句**大字学生原话**（手写感字体）：**"不想把太多时间浪费在安排上。"** 下方小字引用来源：`—— 来自华科大二同学的问卷回答，2026-04`。
- 右侧并列三张小图：①`02_ai_tool_problems.png`；②`05_canvas_scenario_score.png`；③`06_school_distribution.png`。
- 底部一行小字："N=47 · 16 所高校 · 2026-04 实地回收"。

**口播：**

各位评委老师好，我们是 StudySolo。

我想用问卷里收到的一句学生原话开场——**"不想把太多时间浪费在安排上。"**  
这句话道出了当下 AI 学习的真相：**学生不缺聪明的 AI，他们缺的是被安排好的学习路径**。我们在华科、中南财大、华师、武大等 **16 所高校回收了 47 份一线调研问卷**，数据给出了三个清晰信号：

**信号一：AI 已经不是未来，是现在**。**97.9% 的同学正在用 AI 学习**，但痛点高度集中——**55% 说"一次只能问一个问题，没法系统化学习"**；**55% 说"回答经常出错"**；**49% 说"无法保存和复用之前的对话"**。**问题不是 AI 不聪明，而是流程断裂、结果易失**。

**信号二：学生要的是路径和沉淀**。**60% 说"学习资料太多太杂，不知道看什么"**，**55% 说"不知道从哪里开始学"**，**55% 说"学完就忘"**。学习真正缺的是**"编排 + 可视 + 复用"**，而不是又一个聊天框。

**信号三：用户对"可视化学习工作流"认可度极高**。当我们向他们描述"AI 在画布上自动规划、搜索、总结、出题，过程可见、步骤可调"的场景——**85% 给出 4 或 5 分的正向评分，43% 主动留下了内测联系方式**。

这三组数据直接推导出 StudySolo 的核心命题——**把学习变成可编排的流水线**。

## P2 · 我们做什么 + 凭什么是我们（约 50s）

**【PPT 提示】**：上半页一句话定位；下半页**二维象限图**——X 轴（通用 ↔ 垂直学习），Y 轴（单轮对话 ↔ 可编排工作流）。ChatGPT/Kimi 在左下；Dify/Coze 在右上偏通用；Notion AI 在左上；**StudySolo 落在"垂直学习 × 可编排工作流"的空白象限**。

**口播：**

一句话定位：**用户用自然语言描述学习选题，系统在画布生成并迭代学习工作流，按依赖自动执行，过程全程可见，成果持续沉淀复用**。

放到竞品坐标系里看得很清楚：  
和 ChatGPT、豆包这类通用聊天工具比，我们把**单轮问答升级成多步可编排的 DAG**；  
和 Dify、Coze 这类通用工作流平台比，我们有**学习场景专属节点**——闪卡、测验、导图、报告不是通用工具能给的；  
和手工拼工具比，我们在**一张画布里串起检索、分析、生成、测验、导出**。  
更关键的是：我们的 AI **不只用来建图**，还支持持续对话和学习打卡；工作流可保存、可追踪、可 Fork，**学习过程本身成为资产**。

## P3 · 架构：三层讲清楚（约 90s）

**【PPT 提示】**：主图是分层架构图（意图层 → 编排层 → 执行与路由层），右侧小字列技术栈；**左下角嵌入一张诊断面板截图**（`21/23 项通过 · 总耗时 842ms` 这种健康总览）——视觉冲击力拉满。

**口播：**

整个系统可以概括为三层。

**第一层，意图理解与交互层**。前端用可视化画布承载学习流程；每次对话都会把当前画布结构序列化给后端，后端先做意图分类，再决定是聊天、规划还是直接改图，空画布可一键生成整图。

**第二层，DAG 编排与执行引擎**。这一层最直观的类比是**地铁调度中心**——每一个节点就像一列正在运行的车，它的位置、状态、实时产生的 Token，都会通过**六类 SSE 事件**持续回传给前端；前端据此做到**节点级流式输出 + 全局级执行追踪的双层可视化**，同层无依赖节点还会并发执行。这也是问卷里"AI 思考像黑盒"这一痛点的直接回应——**你的每一步学习都看得见**。

**第三层，多模型路由与治理层**——这一层有**三件我们特别要讲的工程能力**。

**一是多模型容灾降级**。每种任务类型都有一条按优先级排好的候选模型链路，主力模型超时或报错会自动切到备选；流式场景下如果已经输出了 Token 就保持一致性不切换，而不是让用户看到"半句话又换模型"。**每一次 fallback、每一次失败都写进计费账本**——这是问卷里 55% 同学吐槽"AI 回答经常出错"的根本应对。

**二是一键诊断 × 一键节点测试——"两个一键"**。管理后台点一次就并行拨测所有组件——数据库、每一个启用的模型 SKU、每一个子 Agent、Embedding 服务，都有真实延迟、错误详情、Markdown / JSON 报告一键导出；与之并列的是**节点级单测入口**，不必跑完整工作流就能回归测试任一节点——**能诊断、能回归，才是能被运维的系统，而不是 Demo**。

**三是合规与数据治理**。服务条款、隐私政策、Cookie 政策三份面向学生场景的合规文档齐备；底层 Supabase RLS 做多租户数据隔离，后台审计日志留痕——这是进校园、进机构的门槛线。

## P5 · 创新点与壁垒（约 55s）

**【PPT 提示】**：三个大图标（垂直 DAG / 开放扩展 / 双沉淀）+ 底部一句"飞轮"。

**口播：**

我们把原本五点壁垒收敛为**三条护城河**。

**第一条是场景护城河——垂直学习 DAG**。我们不是通用聊天，也不是通用工作流：用户从一个学习选题出发，平台直接生成工作流、执行、自测、沉淀，**一条龙对应问卷里最高频的期望："72% 的同学希望 AI 帮他们总结重点，57% 希望自动生成完整学习计划"**——我们把这些都做成了节点，而不是 Prompt。

**第二条是能力护城河——开放扩展 × 工程化双引擎**。对外是**三种扩展能力**：MCP 协议、Skills 技能注入、子 Agent 节点。举例：做《湖北文化旅游》课题的同学，可以在画布里挂一个 **OpenClaw MCP 节点**抓外部产业数据，可以挂 Skills 注入"湖北方言校对"，也可以直接调用已落地的**代码评审 / 学习辅导 / 可视化站点三个子 Agent**——**全部在一张学习工作流画布上完成**。

对内是我们**独创的工程化方法论**——「**SOP 文档 + AI Skills 规则**」双轨协作。我们把节点新增、团队分支提交、系统自检、架构认知这四件事沉淀成了 **4 个本项目原创的 AI Skills**，托管在 `.agent/skills/` 公开开源，**让 AI 在辅助开发时自动遵循我们项目专有的规范**——这在我们了解的学生和工程团队里都是首次实践。**对外扩展让平台成为连接器，对内规范让开发成为流水线**——这是通用工作流平台做不到的双重护城河。

**第三条是数据护城河——过程与结果双沉淀**。执行过程通过 SSE 全程可观测，执行完还能追日志；结果不是一段文字，而是**可直接使用的学习成果**——闪卡、测验、导图、报告——可入库、可导出、可 Fork。更关键的是，**每一次测验作答结果都会回流到用户画像，下一次生成工作流时会根据薄弱点自动调整节点权重**——这是我们和通用工作流最根本的差异：**学习数据驱动工作流进化**。

三条护城河形成飞轮：场景越聚焦，工作流越好用；能力越开放，第三方越愿意接入；数据越厚，个性化越精准。

## 附录 H · 3 分钟视频脚本对照表（v3.5 新增）

> **策略**：3 分钟视频是**补充凭证**而不是"现场答辩压缩版"。视频重点演示**现场不方便做或做不快的部分**——一键启动、合规展示、SOP/Skills 亮点、精确演示耗时；现场答辩则聚焦**技术叙事和评委互动**。

### H.1 视频内容分配建议（总时长 180s）

| 时段 | 内容 | 为什么放视频不放现场 |
|------|------|-----------------|
| **0:00–0:15** | 一句 Slogan 大字 + 域名 + GitHub 仓库地址 | 视频必须前 5 秒抓住观众 |
| **0:15–0:30** | **一键启动脚本演示**：终端敲 `scripts/start-studysolo.ps1` → 前后端 + Agents 全起 | 现场做有风险且耗时；视频里 15 秒搞定 |
| **0:30–1:20** | **完整期末复习流演示**（录屏压缩到 50 秒，加计时器水印） | 现场演示 P4 不到 1 分钟讲不透；视频里可快进到关键节点 |
| **1:20–1:45** | **后台管理面板**：一键诊断 + 一键节点测试双入口 + 导出 Markdown 报告 | 现场切后台风险高，视频直接展示成品 |
| **1:45–2:10** | **SOP + Skills 工程化**：展示 `.agent/skills/` 四个原创 Skill 文件 + AI 辅助开发场景录屏 | 这是口播讲 15 秒容易听糊，视频展示文件结构更直观 |
| **2:10–2:30** | **合规文档三件套**：服务条款 / 隐私政策 / Cookie 政策页面快速切屏 | 现场讲 5 秒带过即可，视频留凭证 |
| **2:30–2:50** | **开源仓库全景**：GitHub 双仓（冻结版 + 开发版）+ Issue / Milestone + Contributors | 证据类内容，视频里一目了然 |
| **2:50–3:00** | Slogan 再重复一次 + 域名 + 二维码 | 视频结尾必须留记忆点 |

### H.2 现场答辩与视频分工原则

| 内容 | 主现场 | 主视频 | 理由 |
|------|-------|-------|------|
| 问卷调研数据 | ✅ | — | 情绪钩子，必须现场讲 |
| 三层架构 | ✅ | 部分图 | 技术叙事主轴，现场讲才有感染力 |
| **一键启动** | — | ✅ | 现场做有风险 |
| 演示期末复习流 | 降级版 | ✅ **主场** | 现场演示风险高，视频作为主证据 |
| 一键诊断 + 节点测试 | 提一句 | ✅ | 现场切后台有风险 |
| **SOP + Skills 独创** | ✅ | ✅ | 现场必讲（技术亮点）；视频展示文件结构 |
| MCP / OpenClaw 例子 | ✅ | — | 现场讲故事性强 |
| 学习闭环 + 商业骨架 | ✅ | — | 现场必讲 |
| 合规文档 | 一句带过 | ✅ | 视频凭证即可 |
| 开源仓库 / 一键启动 | 收尾一句 | ✅ **重点** | 视频做证据留痕 |
| 路线图 | ✅ | — | 现场必讲，对齐 README |
| 团队 + 协作规范 | ✅ | 部分 | 现场讲协作机制；视频可穿插 commit 历史截图 |

### H.3 视频制作要点

- **字幕必备**：评委可能静音看或倍速看，全程字幕
- **计时器水印**：演示段右上角贴精确耗时计时器
- **键盘声 + 鼠标点击音**：增强真实感，证明不是伪造录屏
- **章节标记**：Bilibili / YouTube 上传时打上时间戳章节
- **结尾二维码**：指向 GitHub 仓库（非域名——仓库更体现工程深度）
