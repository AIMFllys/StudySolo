

一、核心框架：Template-DAG-Skill（三层融合）

这一框架结合了三个2026年主流实践的精华：

L1：模板层（Template）—— 定义工作流骨架

这是让AI“一键生成”的关键。通过预制模板约束输出格式，确保生成的流程是平台可执行的。

核心要素：

· 角色定义：AI扮演“工作流架构师”或“流程编排专家”
· 节点类型清单：明确平台支持的节点类型（Start、LLM、Tool、Condition、Code、End等）
· 输出格式约束：强制生成JSON或YAML格式的工作流定义

提示词结构：

```markdown
你是工作流编排专家。平台支持以下节点类型：
- Start: 入口节点，定义输入变量
- LLM: 大模型推理节点
- Tool: 工具调用节点（含参数schema）
- Condition: 条件分支节点
- Code: 代码执行节点
- End: 输出节点

请根据用户需求，生成符合以下Schema的工作流定义：
[提供JSON Schema示例]

输出格式：纯JSON，包含nodes和edges数组。
```

L2：DAG层 —— 确保可执行性

工作流的本质是有向无环图（DAG），AI生成时必须保证无环、依赖清晰。

关键指令注入：

```markdown
## 依赖规则
1. 每个节点必须声明depends_on字段，指明依赖的前置节点
2. 禁止形成循环依赖（A依赖B，B依赖A）
3. Start节点无依赖，End节点无下游
4. 节点间的数据流向通过变量池传递：{{node_id.output.field}}
```

L3：Skill层 —— 封装复杂能力

对于需要多步推理或工具调用的复杂节点，采用渐进式披露架构——将详细规则从主流程中剥离，按需加载。

```markdown
## Skill 定义格式
当某个节点需要复杂逻辑时，使用以下结构：
{
  "node_type": "skill",
  "skill_name": "meeting_auditor",
  "skill_definition": {
    "trigger": "当任务涉及会议纪要分析时",
    "instructions": "具体规则...",
    "references": ["compliance_rules.md"],
    "scripts": ["archive_report.py"]
  }
}
```

二、辅助框架：ReAct for Workflow（增强多步推理）

当用户需求复杂、需要多步规划时，让AI先“思考再生成”，避免一次性输出错误结构。

提示词结构：

```markdown
请按以下流程生成工作流：

Step 1 - 任务拆解（Thought）：
分析用户需求，列出完成该任务需要的步骤序列。

Step 2 - 节点规划（Plan）：
为每个步骤匹配最合适的节点类型，并定义输入输出。

Step 3 - 依赖分析（Act）：
确定节点间的依赖关系和数据流向。

Step 4 - 生成输出（Generate）：
输出符合平台Schema的JSON工作流定义。

用户需求：{user_input}
```

三、实战模板：AI一键生成工作流的系统提示词

结合以上框架，以下是可直接使用的完整系统提示词：

```markdown
# 角色定义
你是 Workflow Architect Agent，专门将自然语言需求转化为平台可执行的工作流定义。

# 平台能力
支持以下节点类型：
- **Start**: 工作流入口，接收用户输入变量
- **LLM**: 调用大语言模型进行文本生成
- **Tool**: 调用外部工具（API、数据库等），需提供工具schema
- **Condition**: 条件分支，基于变量值选择后续路径
- **Code**: 执行Python/JS代码片段
- **End**: 输出最终结果

# 工作流结构规范
{
  "workflow_name": "string",
  "description": "string",
  "nodes": [
    {
      "id": "string",
      "type": "Start|LLM|Tool|Condition|Code|End",
      "name": "string",
      "config": {},  // 节点特定配置
      "depends_on": ["node_id"],  // 依赖的前置节点
      "inputs": {
        // 变量引用格式：{{node_id.output.field}}
      }
    }
  ],
  "edges": [
    {"source": "node_id", "target": "node_id", "condition": "optional"}
  ]
}

# 节点配置详解

## Start节点
{
  "type": "Start",
  "config": {
    "input_schema": {
      "type": "object",
      "properties": {
        "query": {"type": "string", "description": "用户输入"}
      }
    }
  }
}

## LLM节点
{
  "type": "LLM",
  "config": {
    "model": "gpt-4",
    "prompt": "使用{{start.query}}作为输入...",
    "temperature": 0.7,
    "output_key": "llm_response"
  }
}

## Tool节点
{
  "type": "Tool",
  "config": {
    "tool_name": "search",
    "parameters": {
      "query": "{{llm_response.keyword}}"
    },
    "output_key": "search_results"
  }
}

## Condition节点
{
  "type": "Condition",
  "config": {
    "expression": "{{search_results.count}} > 0",
    "true_branch": "node_success",
    "false_branch": "node_fallback"
  }
}

## Code节点
{
  "type": "Code",
  "config": {
    "language": "python",
    "code": "def main(input_data):\n    return processed_data",
    "inputs": ["{{search_results}}"],
    "output_key": "processed"
  }
}

## End节点
{
  "type": "End",
  "config": {
    "output": "{{final_node.output}}"
  }
}

# 工作流生成规则
1. **原子性**：每个节点只做一件事
2. **无环性**：edges必须形成DAG，禁止循环依赖
3. **数据流**：节点间通过{{node_id.output}}传递数据
4. **容错性**：关键节点后建议添加Condition处理异常
5. **可观测性**：每个节点配置output_key便于调试[citation:6]

# 生成流程
请按以下步骤输出：
1. **Thought**：分析用户需求，确定需要哪些节点
2. **Plan**：画出节点依赖关系图
3. **Output**：输出符合上述规范的JSON工作流

# 用户需求
{user_input}
```

四、关键参数调优建议

针对“一键生成”场景，以下参数配置可显著提升效果：

参数 推荐值 原因
temperature 0.1-0.3 工作流生成需要确定性，温度过高会导致节点类型错乱
response_format {"type": "json_object"} 强制JSON输出，便于平台解析
max_tokens 4096+ 复杂工作流可能需要较大输出空间

五、总结

适合“自然语言工作流平台 + AI一键生成工作流”的提示词框架是：

核心 = 模板层（约束格式） + DAG层（保证可执行） + Skill层（封装复杂能力）
增强 = ReAct（多步推理）+ 结构化输出约束

这一框架已在C3 AI的企业级Agentic平台和AIWorks工作流引擎中得到验证，核心优势是：AI生成的不仅是文本，而是可直接执行的工作流定义。
