# 项目规划映射与选型

## 一、从项目 docs 提取到的 AI API 事实

> 🔗 **全局规划文档**：`docs/Plans/daily_plan/API/README.md`

明确规划：
- 使用 `openai` Python SDK 作为统一调用层。
- 后端以 SSE 推送逐步执行结果，前端使用 `EventSource` 接收。
- 分解为 **八大模型供应商**，均走 OpenAI 兼容格式：
  1. 阿里云百炼 DashScope (Native)
  2. DeepSeek (Native)
  3. 月之暗面 Kimi (Native)
  4. 火山引擎 Volcengine (Native)
  5. 智谱AI Zhipu (Native)
  6. 七牛云 QNAIGC (Proxy)
  7. 优云智算 Compshare (Proxy)
  8. 硅基流动 SiliconFlow (Proxy)

## 二、接口优先级（规划阶段）

1. **第一优先：OpenAI Chat Completions 兼容接口**
   - 原因：四家供应商兼容成熟、迁移成本最低。
   - 四家 Base URL 汇总：
     | 平台 | Base URL |
     |------|----------|
     | 阿里云百炼 | `https://dashscope.aliyuncs.com/compatible-mode/v1` |
     | DeepSeek | `https://api.deepseek.com/v1` |
     | Kimi | `https://api.moonshot.cn/v1` |
     | 火山引擎 | `https://ark.cn-beijing.volces.com/api/v3` |
     | 智谱 AI | `https://open.bigmodel.cn/api/paas/v4` |
     | 七牛云 | `https://api.qnaigc.com/v1` |
     | 优云智算 | `https://api.compshare.cn/v1` |
     | 硅基流动 | `https://api.siliconflow.cn/v1` |

2. **第二优先：OpenAI Responses 接口（按需）**
   - 原因：适合后续 agent 化能力（工具调用、上下文延续）。
   - 说明：先保证 MVP 跑通，再逐步迁移到 Responses。
   - 备注：百炼已声明兼容 Responses API；七牛云也在跟进。

3. **Embedding 接口（为语义检索预留）**
   - 原因：项目规划已出现向量检索需求（pgvector / embedding 字段）。
   - 备注：百炼已提供 OpenAI 兼容的 Embedding 接口。

## 三、统一环境变量

在 `.env` 中，为八个供应商配置必需项（部分已在 SDK 隐藏 Base URL，这里为了全平台调度也做提取）：

```env
# 原生直连层
DASHSCOPE_API_KEY=
DEEPSEEK_API_KEY=
MOONSHOT_API_KEY=
VOLCENGINE_API_KEY=
ZHIPU_API_KEY=

# 代理层
QINIU_AI_API_KEY=
COMPSHARE_API_KEY=
SILICONFLOW_API_KEY=
```
