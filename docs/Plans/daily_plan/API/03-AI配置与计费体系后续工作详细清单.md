## 2. 当前基线状态

截至 2026-03-26，项目已经完成以下核心升级：

- 运行时配置已切为“YAML 路由策略中心”
- 模型目录与价格元数据已切为“Supabase family + SKU 双层目录”
- usage 账本已切为“平台级 SKU + CNY 价格快照”
- 用户侧选模已切到 catalog API
- 后台模型管理已切到 catalog API
- 左侧个人面板与后台 AI 仪表盘已统一为 CNY 口径
- assistant / workflow 双账本已完成分离
- 规范文档、API 规范、命名规范、新增模型 SOP 已同步建立

当前业务策略已经明确，不再作为待讨论事项：

- 便宜模型：原生 API 优先
- 贵模型：`qiniu -> siliconflow -> native`
- 工具模型：能力固定路由
- OCR：智谱原生优先
- 搜索：七牛云主通道，预留智谱搜索扩量能力
- 计费按“实际调用平台 + 实际模型 SKU + 实际价格快照”结算

## 3. 后续工作的总体目标

后续阶段不再重做底层架构，重点转为：

- 补齐价格与平台元数据
- 做真实业务链路验收
- 收缩旧兼容层
- 细化路由策略
- 提高测试覆盖率
- 建立可持续维护机制

一句话概括：  
从“已经能跑的统一体系”推进到“价格完整、验证充分、维护清晰、可持续运营”的正式状态。

## 4. 工作流拆分

后续工作建议拆成 6 个并行关注的工作流。

### 4.1 工作流 A：价格与模型目录补全

这是当前最优先的未完成项。

#### 目标

让 `ai_model_skus` 中所有正式启用的 SKU 都具备可靠价格，避免出现：

- 有调用
- 有 token
- 金额却是 `0 CNY`

#### 需要做的事

1. 逐个核查仍为 `Pricing pending` 的 SKU
2. 按平台而不是按 vendor 补价格
3. 为每个价格补充来源链接或来源说明
4. 写入 `pricing_verified_at`
5. 标注是否属于：
   - `native`
   - `proxy`
   - `tool_service`
6. 明确哪些 SKU 只是 fallback，不应该前台可选

#### 优先补齐对象

第一梯队：

- Moonshot 各通道模型
- GLM 各通道模型
- Doubao 现有映射模型
- SiliconFlow 代理的高价模型
- 七牛云搜索相关计费映射

第二梯队：

- 后续准备接入但尚未正式启用的海外高价模型
- 工具类扩展模型

#### 验收标准

- 所有 `is_enabled = true` 且 `is_user_selectable = true` 的 SKU 均有明确价格
- 所有 `proxy_first` 链上会被命中的 SKU 均有明确价格
- 后台模型目录页不再大面积出现 `Pricing pending`
- 同一 vendor 跨 provider 的价格差异能够在后台看出来

### 4.2 工作流 B：真实链路联调与人工验收

当前已经完成了结构层和服务层验证，但还需要补“真实页面 + 真实请求 + 真实数据库写入 + 真实看板刷新”的联调。

#### 目标

确认整条链路在用户实际操作下表现正常：

- 前端选模正常
- 请求进账正常
- provider 调用明细正常
- cost 正常
- 左侧面板更新正常
- 后台仪表盘更新正常

#### 必做验收路径

1. 用户侧聊天
   - 从 catalog 选择一个便宜模型
   - 发起一次 chat
   - 检查 `selected_model_key` 是否正确传输
   - 检查 usage event 是否写入正确 `provider / vendor / sku_id`

2. 用户侧生成工作流
   - 触发一次 `generate-workflow`
   - 确认该请求记入 `assistant`
   - 不得误计入 `workflow`

3. 工作流执行
   - 运行一个包含 AI 节点的 workflow
   - 确认每个 AI 节点都有 usage event
   - 确认逻辑请求记为 `workflow`

4. fallback 场景
   - 构造主通道失败
   - 观察是否正确进入 fallback
   - 校验 `provider_call_count`、`fallback_rate` 和 recent calls

5. 工具类调用
   - OCR 场景验证走智谱原生
   - 搜索场景验证走七牛云主通道

#### 验收标准

- 15 秒轮询内左侧 `DashboardPanel` 出现新数据
- 15 秒轮询内后台 AI 仪表盘出现新数据
- usage 明细、分钟聚合、概览 totals 三者一致
- assistant / workflow 分账无串账
- fallback 请求不会污染逻辑请求数

### 4.3 工作流 C：旧兼容层收缩

当前为了平稳迁移，系统保留了多处兼容入口。后续稳定后，需要逐步减少这些历史包袱。

#### 目标

将系统最终收口到“新目录 + 新字段 + 新接口”。

#### 当前兼容层

- 旧请求字段：`selected_platform + selected_model`
- 旧目录表：`public.ai_models`
- 前端 fallback 常量模型列表
- 部分旧环境变量别名

#### 后续处理方式

第一阶段：

- 保持兼容，但增加日志观察使用频率
- 明确哪些调用仍在走旧字段
- 明确哪些地方仍依赖 `ai_models`

第二阶段：

- 前端默认只传 `selected_model_key`
- 管理后台不再展示旧目录概念
- 新文档只保留新字段，不再继续写旧字段为主路径

第三阶段：

- 删除对旧请求字段的依赖
- 将 `public.ai_models` 从“兼容可读”降为“仅历史保留”
- 清理前端常量作为正式数据源的残余逻辑

#### 验收标准

- 日常主链路全部只用 `selected_model_key`
- 新增模型流程不再要求更新 `public.ai_models`
- catalog API 成为前后端唯一正式目录来源

### 4.4 工作流 D：路由策略细化

当前路由体系已经完成第一版收口，但后续还需要继续细化，使其真正贴近业务策略。

#### 目标

让 YAML 中的 `task_routes` 真正表达：

- 便宜模型原生优先
- 贵模型聚合优先
- 工具模型能力固定

#### 需要继续做的事

1. 梳理所有任务族
   - `cheap_chat`
   - `premium_chat`
   - `reasoning`
   - `strict_json`
   - `workflow_runtime`
   - `ocr`
   - `search`
   - 后续可能的 `embedding`、`rerank`

2. 为每个任务族明确默认链
3. 将高价模型链明确成：
   - `qiniu -> siliconflow -> native`
4. 明确“仅 fallback 可用”的 SKU
5. 补齐 `tool_service` 路由表达
6. 预留 `search + zhipu 扩量` 的复合能力表达

#### 额外需要考虑的点

- 是否需要引入 `compshare` / 优云智算作为特定链路
- 是否需要区分国内主链与海外备链
- 是否需要为“高稳定 JSON 输出”单独设链

#### 验收标准

- `config.yaml` 中每类任务都能映射到清晰 route
- 不再出现“便宜模型误走代理主通道”
- 不再出现“高价模型误走原生主通道”
- OCR、搜索不混入普通聊天链

### 4.5 工作流 E：测试补强

当前已有基础测试，但覆盖率仍不足以支撑长期演进。

#### 目标

把当前系统从“改完能跑”提高到“后续继续改也不容易回退”。

#### 后端建议补的测试

- `selected_model_key` 归一化测试
- catalog service 缓存命中/失效测试
- 同 vendor 跨 provider 价格结算测试
- `proxy_first` 路由顺序测试
- OCR / 搜索固定路由测试
- 价格快照写账测试
- 旧字段兼容入口测试

#### 前端建议补的测试

- catalog API 驱动的选模器渲染测试
- `DashboardPanel` 的 CNY 字段展示测试
- 后台模型目录编辑页行为测试
- 后台 AI 仪表盘 provider / billing_channel 维度渲染测试
- 兼容旧字段时的请求构造测试

#### 验收标准

- 核心 catalog、route、billing、dashboard 逻辑均有自动化覆盖
- 后续新增模型不会轻易打坏账本口径

### 4.6 工作流 F：运营与维护机制

这一部分不是写代码，而是为了避免未来再次失控。

#### 目标

让 AI 平台目录、价格、路由和计费成为“有人负责、可持续维护”的系统。

#### 建议建立的机制

1. 价格核验周期
   - 建议按周或双周核验高频 SKU
   - 至少核验高价模型和用户常用模型

2. 新增模型准入流程
   - 必须走 [新增AI模型-SOP.md](/D:/project/Study_1037Solo/StudySolo/docs/项目规范与框架流程/功能流程/新增AI模型/新增AI模型-SOP.md)

3. 路由变更审查
   - 修改 `task_routes` 时必须说明成本影响
   - 修改高价模型 fallback 链时必须说明容灾逻辑

4. 看板巡检
   - 周期性检查 fallback rate、error rate、cost 波动
   - 观察是否有某个代理平台成本异常偏高

5. 文档同步规则
   - 任何影响 provider / vendor / model_key / billing_channel 的改动，都要同步更新规范文档

#### 验收标准

- 新增模型时不再需要临时口头解释
- 路由变更可追踪
- 价格来源可追溯
- 成本异常有面板可观察

## 5. 建议执行顺序

建议严格按以下顺序推进。

### 第一阶段：先把数据补齐

1. 补 `ai_model_skus` 价格
2. 补 `price_source`
3. 补 `pricing_verified_at`
4. 确认启用 SKU 没有大量 `0 CNY`

原因：

- 没有完整价格，后续所有计费验证都会失真

### 第二阶段：做真实联调

1. 聊天
2. 生成工作流
3. 执行工作流
4. fallback
5. OCR / 搜索

原因：

- 先验证“整条链路正确”，再决定是否可以删兼容层

### 第三阶段：细化 route

1. 明确 cheap / premium / tool 三大类 route
2. 明确各任务族 task family
3. 调整 YAML 表达

原因：

- 路由细化必须建立在真实联调和价格完整的基础上

### 第四阶段：收缩兼容层

1. 缩旧请求字段
2. 弱化 `public.ai_models`
3. 清理旧常量依赖

原因：

- 兼容层删早了容易影响现网稳定

### 第五阶段：补齐自动化测试和运营机制

1. 测试补强
2. 巡检规则
3. 周期性价格核验

## 6. 当前明确未完成项

以下内容截至当前仍视为未完成：

- 部分平台 SKU 价格尚未补全
- 真实 UI 全链路人工联调尚未做完
- 旧兼容层尚未收缩完毕
- route 仍有继续细化空间
- catalog / billing / dashboard 自动化测试仍需继续补齐
- 价格核验周期和日常巡检机制尚未正式落地

## 7. 后续实施时的硬约束

后续继续改造时，必须遵守以下约束：

- 不得把运行时核心路由完全迁移到后台可编辑配置
- 不得重新把价格逻辑写回 YAML
- 不得按 vendor 模糊结算成本
- 不得让工具模型混入普通聊天路由链
- 不得新增仅文档存在、数据库未建档的正式可用模型
- 不得绕过 `selected_model_key` 再扩展新的正式请求字段

## 8. 一句话结论

接下来真正需要做的，不是再推翻这套架构，而是把“价格完整度、真实联调、兼容层收缩、路由细化、测试覆盖、运营机制”六件事逐步做扎实。完成这些之后，StudySolo 的 AI 配置、调用、计费和后台分析体系才算真正进入稳定运营阶段。
