# 用户反馈通道与奖励系统全栈实现总结

**创建时间**: 2026-03-25
**功能模块**: 前端设置面板的反馈抽屉 / 后端反馈入库与激励分发 / 数据库表结构定义

## 1. 核心目标
为了收集用户对 StudySolo 的真实评价与 bug 反馈，在应用的左下角提供了一个高保真 “学术手绘笔记本” 审美风格的**反馈信箱入口**。用户提交反馈后，平台会在后端自动记录其问卷数据，并为优质反馈者自动追加 X 天的 Pro 高级会员时长，作为**反馈激励系统**。

---

## 2. 前端实现：纸质审美的反馈抽屉 (Drawer)

**路径**: `frontend/src/features/settings/components/FeedbackChannel.tsx`

### UI / UX 设计
1. **入口设定**：放置在左侧导航栏设置页最下方的一键开启按钮，清晰且低侵入性。
2. **复刻纸质手账审美**：
   - 移除厚重的纯色填充，采用 `node-paper-bg` 配合极为微弱的半透明背景呈现类似牛皮/象牙白纸张的基调。
   - 通过 SVG 画布构建细致的浅色背景网格（Scholarly Grid），并配合绝对定位画出经典的“双重红色左侧边距线”。
3. **沉浸式交互层**：
   - 借助 `framer-motion` 实现右侧丝滑平移划出的抽屉式模态框。
   - 借助 `createPortal` 挂载至 `document.body` 确保不被外层元素遮挡。
4. **统一收拢的操作栏**：
   - 问卷最底部不再悬浮脱离视图，而是封装在带有 `ShieldCheck` (盾牌安全认证图标) 的优雅背景卡片中，将“暂不提交”和“立即提交”操作妥帖收纳，防止了视觉突兀。

## 3. 数据库实现：`ss_feedback`

**路径**: `supabase/migrations/20260325072500_create_ss_feedback_table.sql`

创建了 StudySolo 专属的反馈数据库表（与现有其它产品表隔离）：
```sql
CREATE TABLE IF NOT EXISTS ss_feedback (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  user_email    text NOT NULL DEFAULT '',
  user_nickname text DEFAULT '',
  rating        smallint NOT NULL CHECK (rating BETWEEN 1 AND 5),
  issue_type    text DEFAULT '',
  content       text NOT NULL DEFAULT '',
  reward_days   smallint NOT NULL DEFAULT 3,
  reward_applied boolean NOT NULL DEFAULT false,
  created_at    timestamptz NOT NULL DEFAULT now(),
  updated_at    timestamptz NOT NULL DEFAULT now()
);
```

**安全性设计**: 
启用了 RLS（行级安全性），配置以下策略确保数据隔离与安全：
- 仅允许经过认证的用户 (auth.uid() = user_id) 插入反馈数据。
- 仅允许用户读取自己的历史反馈。
- Service Role（后端管理级 Key）拥有一切权限。

## 4. 后端实现：反馈收集与会员激励分发

**路径**: `backend/app/api/feedback.py` 并且注册于 `backend/app/api/router.py`

### 核心接口逻辑 (`POST /api/feedback`)
1. 提取 FastAPI 解析的 Header Token 并通过共享的 `get_current_user` 获取真实用户身份。
2. 聚合获取 `user_profiles` 里的冗余信息（比如 `nickname`）录入反馈表，减少后续多表查查询。
3. **会员奖励系统 (`_grant_membership_reward`)**：
   - 读取当前用户的订阅状态 (`tier`) 和到期时间 (`tier_expires_at`)。
   - 若用户目前为免费会员，则立刻变为 `pro` 并以当前时间叠加 N 天有效期。
   - 若用户目前已经是高等级 `pro`，则智能追加在他们的现有生命周期结束时间之上。
4. 提供 `/mine` 接口用于后续拓展客户端展示用户历史反馈及累计获取奖励的积分大盘。

## 5. 项目架构与规范合规性校验结论
- [x] **行数限制**：`FeedbackChannel.tsx` 及 `feedback.py` 均控制在约 ~250 行左右，满足不超过 300 行的架构规定，过度复杂的逻辑均提取为内联独立函数。
- [x] **美学一致性**：摒弃了通用的极客玻璃拟态风，对齐了 “Ink & Parchment” 和“学术蓝图”（Scholarly Blueprint）的美学体系要求。
- [x] **统一路由流**：前端沿用 `authedFetch` 自带无感刷新 Token 的能力；后端整合在 `APIRouter()` 下统一下发授权。 
- [x] **开发完整度**：从 SQL 设计到 API 落地再到前端展现交互，全栈闭环实现。
